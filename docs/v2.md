# LBA v2 稳定说明

LBA v2 把顶层契约收敛成一个直接从 dataset 构造的 DataLoader：

```python
from lba import LBA


loader = LBA(
    dataset,
    len_fn=sample_length,
    batch_size=32,
    shuffle=True,
    collate_fn=collate_fn,
    max_padded_length=8192,
)
```

这是破坏性主版本升级。旧的 loader-wrapper 构造方式和多个公开 adapter 类名不再属于
公共接口，也不提供静默兼容。入口错误会直接暴露，避免配置表面生效但 sampler 实际
没有进入 source loader。

## v2 对外契约

- `LBA` 是唯一公开 loader 类，继承 `torch.utils.data.DataLoader`。
- 第一个参数是 dataset；`len_fn` 是必填 keyword-only 参数。
- LBA 接受标准 DataLoader 参数和自身 planner 参数，并遵守 PyTorch 参数互斥规则。
- `len_fn` 接收 raw dataset sample，在最终 `collate_fn` 之前运行。
- 最终动态 batch 继续由用户提供的 `collate_fn` 构造。
- 动态 batch 数无法提前准确计算，因此 `len(loader)` 明确不可用。
- `max_padded_length` 约束普通 planned batch；超预算 sample 会作为 singleton 输出并
  warning。
- `max_padding_ratio` 是 fast-path threshold，不保证 fallback 或 flush batch 都低于
  该值。
- 默认 quality planner 保留代表候选 fallback search，优先保持低 padding。
- LBA 不承诺精确 sample 顺序，也不承诺不同 planner 模式产生相同 batch 边界。
- 普通非 DDP iteration 只有在完整消费、source 不 drop、没有 `max_batches` 时才承诺
  sample 守恒。

## Lightning 契约

DataLoader 子类身份是 v2 的关键边界。Lightning 可以在 data hook 返回后重建 LBA，
注入 `DistributedSampler`，并在每个 epoch 调用 sampler 的 `set_epoch()`。注入后的
sampler 必须进入 LBA 内部 source loader，不能只停留在外层属性。

调用侧应保持 `use_distributed_sampler=True`，不再为 LBA 手工构造 rank sampler 或
sampler epoch callback。这个自动路径只适用于 map-style dataset；IterableDataset
必须自行分片。

PyTorch `DistributedSampler(drop_last=False)` 会在 dataset size 不能整除 world size
时补齐 index，产生重复样本。v2 保留该 sampler 语义，不做隐式去重。

## DDP 对齐

steady state 中，每个非空 source batch 对应一个 planned batch。final flush 汇总各
rank 尾部 pool，重新规划并分配相同步数，避免 backward collective 次数不同。

map-style dataset 有稳定 index 时只交换 metadata，并在接收 rank 主进程重取 sample；
dataset lookup 必须确定、无副作用、worker 外可用并保持有效长度。IterableDataset 或
缺少 index metadata 的路径使用 object gather，尾部 sample 必须可 pickle。

所有 rank 必须同步消费、同步停止，并使用相同的 budget 与 planner 控制流配置。默认
`drop_last_flush=True` 会丢弃无法组成全 rank 非空 step 的 final tail 并 warning；设为
`False` 时直接报错。

## IterableDataset 边界

IterableDataset 仍通过 `LBA(dataset, ...)` 使用，不另设公共入口。它必须配置 batched
loading，不能使用 `batch_size=None`。dataset 自身负责 distributed rank 和 worker
sharding，并保证每个 rank 的非空 source batch 数一致。

iterable 自己决定 repeatability 和 cursor 语义。one-shot iterator 不会自动重建或
回放；planner lookahead 和 prefetch 可能读取最后一个已输出 batch 之后的 source
items。

## Resume 边界

v2 不 checkpoint source iterator cursor、planner pool 或 prefetched lookahead。
epoch-boundary resume 可以依赖确定性 sampler 的正常语义，但 mid-epoch resume 不保证
精确 sample continuity。

需要 step checkpoint 精确续采的调用侧必须提供 stateful dataset/sampler，并设计
pending planner state 的保存与恢复。恢复 model、optimizer 和 Lightning loop state
不等于恢复 LBA 数据游标。

## Planner 基线

`planner_mode="quality"` 是 v2 默认和推荐基线。`planner_mode="throughput"` 只在真实
训练指标证明 producer 是瓶颈后启用；candidate cap 和 adaptive fallback 会改变
search work 分布，不保证 wall time 或 padding 无条件更优。

v2 改变的是公共构造和框架集成边界，不改写已有 planner benchmark 结论。v2 API 的
新一轮端到端 benchmark 应和历史结果分开记录。
