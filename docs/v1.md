# LBA v1 稳定说明

当前版本是 LBA 的稳定 v1。v1 的默认目标不是追求纯 CPU 迭代速度接近原始
`DataLoader`，而是在真实训练里提供稳定的动态 batch、低 padding、可读日志和明确
的 DDP 行为。

## 推荐用法

最小接入方式仍然是先创建原始 `DataLoader`，再用 `LBA` 包一层：

```python
from lba import LBA


def sample_length(sample):
    return len(sample["input_ids"])


loader = LBA(
    dataloader,
    len_fn=sample_length,
)

for batch in loader:
    train_step(batch)
```

如果模型有明确的 padded length 或 token budget，推荐显式设置：

```python
loader = LBA(
    dataloader,
    len_fn=len_fn,
    max_padded_length=8192,
    planner_mode="quality",
    max_padding_ratio=0.05,
    log_dir="outputs/lba_logs",
)
```

`planner_mode="quality"` 是 v1 默认和推荐基线。只有当训练侧 loader wait / GPU
utilization 与 LBA planner 统计共同表明 producer 是瓶颈时，才建议显式切到
`planner_mode="throughput"` 并重新 benchmark。

## v1 对外契约

- LBA 对外仍然是一个 PyTorch `DataLoader` 风格的 iterable wrapper。
- `len_fn` 接收原始 dataset sample，运行在用户原始 `collate_fn` 之前。
- LBA 选出动态 batch 后，仍然调用用户原始 `collate_fn`。
- `max_padded_length` 约束普通 planned batch；单个 sample 自身超出预算时会作为
  oversized singleton 输出并记录 warning。
- `max_padding_ratio` 是 fast-path 提交阈值，不保证 fallback 或 flush batch 都低于
  该值。
- 默认 quality planner 保留不设上限的代表候选 fallback search，优先保持低 padding。
- DDP final flush 会重新规划各 rank 的剩余样本，让每个 rank 执行相同数量的
  steps；无法组成非空 step 的尾部样本默认丢弃并记录 warning。
- LBA 不承诺动态 batch 的精确样本顺序，也不承诺不同 planner 模式之间产生相同
  batch 边界。
- 默认样本守恒只覆盖完整消费的普通非 DDP iteration；source `drop_last`、
  `max_batches`、提前停止和 DDP final-tail drop 是明确例外。

## 已纳入 v1 的优化

- `BatchPlanner` 维护 sorted records；`CandidateIndex` 为当前 pool 构造 prefix
  lengths 和 sorted lengths，并按需构造 arrival-id range-min view，减少 planner 与
  候选搜索之间的并行数组传递。
- quality planner 使用 recent-window fast path，并保留不设上限的代表候选 fallback search；这能
  降低 steady-state 成本，同时维持默认 padding 质量。
- `Iteration` 持有单次迭代状态和主流程；`DistributedBatchCoordinator` 负责
  collectives、indexed/object 传输选择与 indexed sample materialization，
  `DistributedFlushPlanner` 负责公共尾部池的规划和分配。
- `RunReporter` 统一 summary 与 oversized sample 的人类可读/JSONL 双格式报告；
  wrapper 只保留 adapter 构造、source 入口和 iteration 生命周期衔接，其他子系统记录
  自己拥有的 config、spill 和 DDP 事件。
- `BudgetResolver` 表达长度预算解析；旧的 `LengthBudgetResolver` 名字保留为兼容
  alias。

## 尝试过但未替换默认的优化

缩小候选范围是最直接的吞吐优化。`planner_mode="throughput"` 会限制普通迭代中
recent-window 搜索的候选窗口数量，默认上限是 `256`。它能压住单次 `pop_ready()`
的搜索量，但候选范围过窄时，样本会更容易滞留到 final flush，最终可能增加 batch
数、flush 搜索量或 padding。

只在 capped search 连续失败时做不设上限的代表候选 fallback 也不够。实际 benchmark 里，问题不
总是表现为 `no_ready`，也可能是局部候选持续可出 batch，但 pool 仍然变大，导致尾部
flush 债务累积。

adaptive throughput fallback 因此又增加了两条安全阀：连续 miss 达到阈值后允许
一次不设上限的代表候选搜索；pool 达到阈值后临时取消本次 threshold search 的窗口上限。这确实能
减少 flush 债务，但它把更多搜索工作放回 steady-state，padding 和总耗时也不是稳定
胜过 quality。因此它保留为 throughput 模式的 opt-in 行为，而不是替换 v1 默认。

更激进的长度 bucket、候选缓存和 aging 策略仍然有探索价值，但它们会改变 batch
选择语义和状态维护复杂度，适合以后作为非默认实验模式单独验证。
